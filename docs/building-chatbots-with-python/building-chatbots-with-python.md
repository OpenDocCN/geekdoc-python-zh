## 第1章：备受喜爱的聊天机器人

当你开始构建聊天机器人时，理解聊天机器人的功能和外观非常重要。

你一定听说过Siri、IBM Watson、Google Allo等。这些机器人试图解决的基本问题是成为中介，帮助用户提高生产力。它们通过让用户少担心如何检索信息以及获取特定数据可能需要的输入格式来实现这一点。随着机器人处理用户数据输入并从中获得更多洞察，它们往往会变得越来越智能。聊天机器人之所以成功，是因为它们能准确提供你想要的东西。

当你不得不在不同的网站上每次都输入相同的姓名、电子邮件ID、地址和邮政编码时，这是否会让你感到恼火或沮丧？想象一个单一的机器人来完成你的任务——比如从不同的供应商订购食物、从各种电子商务公司在线购物，或者预订航班或火车票——而你不必每次都提供相同的电子邮件ID、收货地址或支付信息。

机器人已经具备了解这些信息的能力，并且足够智能，当你用你自己的语言或计算机科学中所谓的自然语言向它提问时，它能检索到所需的内容。

聊天机器人的开发比几年前容易得多，但聊天机器人在几十年前就已经存在了；然而，在过去几年中，聊天机器人的受欢迎程度呈指数级增长。

如果你是技术人员或对Web应用程序或移动应用程序的工作原理有所了解，那么你一定听说过API这个术语。你今天需要的任何类型的数据都可以通过不同服务提供商和机构提供的API形式来获取。如果你正在寻找天气信息、预订票务、订购食物、获取航班信息、将一种语言转换为另一种语言，或者在Facebook或Twitter上发帖，所有这些都可以使用API来完成。这些API被基于Web或移动的应用程序用来执行这些任务。聊天机器人也可以使用这些API，根据我们的请求来实现相同的任务。

聊天机器人之所以在在线完成任务方面比传统方法更具优势，是因为你可以借助聊天机器人完成多项任务。它不仅仅是一个聊天机器人，更像是你的虚拟个人助理。你可以想象一下，既能通过它在booking.com上预订酒店房间，又能预订酒店附近餐厅的餐桌，而这一切都可以通过你的聊天机器人来完成。聊天机器人满足了多功能的需求，从而节省了大量的时间和金钱。

在本书中，我们将学习如何使用机器人构建自然的对话体验，以及如何教会机器人理解我们的自然语言，并让它通过单一界面为我们执行任务。

### 聊天机器人的简要时间线

让我们回顾一下聊天机器人发展历程的简要时间线。了解聊天机器人的技术起源及其演变过程非常重要。

聊天机器人近年来确实日益普及，但其背后的努力是基于数十年的技术积累。回顾聊天机器人的历史，你一定会惊叹于我们自起步以来所取得的巨大进步。

#### 1950年

艾伦·图灵提出了图灵测试。这是一项测试机器是否能表现出与人类相当或无法区分的智能行为的测试。

第一个聊天机器人Eliza由约瑟夫·维森鲍姆创建，旨在充当治疗师。它通过使用“模式匹配”和替换方法来模拟对话，给用户造成机器人能够理解的错觉。

#### 1972年

Parry是由精神病学家和斯坦福大学科学家肯尼斯·科尔比编写的一个计算机程序，它模拟了偏执型精神分裂症患者的行为。

#### 1981年

Jabberwocky聊天机器人由英国程序员罗洛·卡彭特创建。它始于1981年，并于1997年在互联网上发布。

这个聊天机器人的目标是“以有趣、娱乐和幽默的方式模拟自然的人类对话。”

#### 1985年

无线机器人玩具Tomy Chatbot会重复其磁带上录制的任何信息。

#### 1992年

Dr. Sbaitso是由Creative Labs为MS-DOS创建的聊天机器人，它以数字化的声音“与用户交谈”，仿佛自己是一位心理学家。用户反复的辱骂和错误的输入会导致Dr. Sbaitso在重置自身之前出现“奇偶校验错误”而“崩溃”。

#### 1995年

A.L.I.C.E（人工智能语言互联网计算机实体）由诺贝尔奖得主理查德·华莱士开发。

#### 1996年

Hex由杰森·哈钦斯开发，基于Eliza，并于1996年获得勒布纳奖。

#### 2001年

Smarterchild是由ActiveBuddy开发的智能机器人，广泛分布在全球即时消息和短信网络中。最初的实现迅速扩展，可即时访问新闻、天气、股票信息、电影放映时间、黄页列表和详细的体育数据，以及各种工具（个人助理、计算器、翻译器等）。

#### 2006年

Watson的想法源于一次餐桌谈话；它被设计用来参加电视节目《危险边缘》。在第一次尝试中，它只能答对约15%的问题，但后来Watson能够定期击败人类参赛者。

#### 2010年

Siri作为iPhone应用推出，随后被整合为iOS的一部分。Siri是SRI国际人工智能中心的衍生产品。其语音识别引擎由Nuance Communications提供，Siri使用先进的机器学习技术来运行。

#### 2012年

谷歌推出了Google Now聊天机器人。它的原始代号是“Majel”，以《星际迷航》系列中计算机系统的配音演员、吉恩·罗登贝瑞的妻子玛杰·巴雷特命名；它也曾被代号为“助手”。

#### 2014年

亚马逊发布了Alexa。“Alexa”这个词中的“X”是一个硬辅音，因此可以被更精确地识别。这是亚马逊选择这个名字的主要原因。

#### 2015年

Cortana是由微软创建的虚拟助手。Cortana可以设置提醒、识别自然语音，并使用来自必应搜索引擎的信息回答问题。它的名字来源于《光环》电子游戏系列中的一个虚构人工智能角色。

-   有助于全面了解当前问题。查看决策树，我们可以轻松理解缺少什么或需要修改什么。
-   有助于更快地调试。决策树就像一本简短的圣经，或者说，是软件需求规格说明文档的可视化表示，开发人员、产品经理或领导层可以参考它来解释预期行为或在需要时进行任何更改。

#### 意图

当用户与聊天机器人交互时，他使用聊天机器人的意图是什么/他在询问什么？

例如，当用户对聊天机器人说“预订电影票”时，我们作为人类可以理解用户想要预订电影票。这就是机器人的意图。它可以被命名为意图。

另一个例子是当用户说“我想点餐”或“你能帮我点餐吗？”这些可以被命名为意图。同样，你可以根据需要定义任意数量的意图。

#### 实体

意图有关于意图的元数据，称为实体。在“预订电影票”的例子中，预订票可以是一个意图，而实体是票的类型，它也可能是其他东西，比如航班、音乐会等。

你可以为整个意图标记通用实体。实体可以表示数量、计数或体积。意图也可以有多个实体。

类别：鞋
尺码：8

话语不过是你的用户可能表现出的同一问题/意图的不同形式。

### 训练机器人

### 词性标注

词性标注是一个过程，在这个过程中，你阅读一些文本，并为每个单词或词元分配词性，例如名词、动词、形容词等。

当你想要识别给定句子中的某个实体时，词性标注变得极其重要。第一步是进行词性标注，看看我们的文本包含什么。

让我们通过一些真实的词性标注示例来实践一下。

#### 示例 1：

```

('am', 'VERB')

('how', 'ADV')
('to', 'PART')
('build', 'VERB')
('chatbots', 'NOUN')
```

#### 示例 2：

### 词干提取与词形还原

词干提取是将屈折变化的单词还原为其词干、基本形式的过程。词干提取算法将单词 “saying” 还原为词根 “say”，而 “presumably” 则变为 presum。如你所见，这可能并不总是 100% 正确。

**词形还原**与词干提取密切相关，但词形还原是基于单词的预期含义确定其词元（lemma）的算法过程。

例如，在英语中，动词 “to walk” 可能以 “walk”、“walked”、“walks” 或 “walking” 的形式出现。基本形式 “walk”，即人们可能在词典中查找的形式，被称为该单词的 *词元*。spaCy 没有内置的词干提取器，因为词形还原被认为更准确、更有效。

词干提取与词形还原的区别

- **词干提取**以一种粗糙的、启发式的方式完成工作，它截断单词的末尾，假设剩下的部分就是我们实际要找的单词，但这通常包括去除派生词缀。

### 词形还原

**词形还原**试图通过使用词汇表和单词的形态分析来更优雅地完成这项工作。它尽力只移除单词的屈折词尾，并返回单词的字典形式，即词元。

尽管很少有库同时提供词干提取和词形还原的方法，但使用词形还原来正确获取词根始终是最佳实践。

让我们通过一些例子来探索词形还原：

#### 示例 1：

```python
from spacy.lemmatizer import Lemmatizer
from spacy.lang.en import LEMMA_INDEX, LEMMA_EXC, LEMMA_RULES

lemmatizer = Lemmatizer(LEMMA_INDEX, LEMMA_EXC, LEMMA_RULES)
lemmatizer('chuckles', 'NOUN') # 2nd param is token's part-of-speech tag
```

**输出：**

```
[u'chuckle']
```

#### 示例 2：

```python
lemmatizer('blazing', 'VERB')
```

**输出：**

```
[u'blaze']
```

#### 示例 3：

```python
lemmatizer('fastest', 'ADJ')
```

**输出：**

```
[u'fast']
```

如果你想看看词干提取器和词形还原器之间的比较，那么你需要安装 Python 最流行的库之一：自然语言工具包（NLTK）。spaCy 最近才流行起来，但正是 NLTK 让每一位 NLP 爱好者投身于 NLP 及其技术的海洋。

查看以下示例，我们尝试使用 NLTK 提供的两种词干提取技术。首先，我们尝试使用 PorterStemmer 获取单词 "fastest" 的词干，然后使用 SnowBallStemmer。两者给出相同的结果——即 "fastest"——但当我们使用 spaCy 的方法进行词形还原时，它给出 "fast" 作为 "fastest" 的词干，这更有意义且正确。

```python
from nltk.stem.porter import *
from nltk.stem.snowball import SnowballStemmer

porter_stemmer = PorterStemmer()

```

在尝试运行此代码之前，请确保使用 pip3 安装 nltk 包。

这就是词形还原发挥作用的地方。想象一下，你用文本 *will the next season of Game of Thrones be* 进行搜索。

现在，假设搜索引擎进行简单的文档词频匹配来给你搜索结果。在这种情况下，上述查询可能无法匹配标题为 *of Thrones next season release date* 的文章。

如果在将输入与文档匹配之前，我们对原始问题进行词形还原，那么我们可能会得到更好的结果。

### 命名实体识别

**命名实体识别**也被称为**实体识别**或**实体**，是一个在给定文本中查找和分类[命名实体](https://example.com)并将其归入预定义类别的过程。

NER 任务在很大程度上依赖于用于训练 NE 提取算法的知识库，因此它可能有效也可能无效，具体取决于其训练所用的数据集。

spaCy 附带了一个非常快速的实体识别模型，能够从给定文档中识别实体短语。实体可以是不同类型，例如人物、地点、组织、日期、数字等。这些实体可以通过 **doc** 对象的 **.ents** 属性访问。

让我们借助 spaCy 强大的 NER 标注能力，通过一些示例来尝试查找命名实体。

#### 示例 1：

```python
my_string = u"Google has its headquarters in Mountain View, California having revenue amounted to 109.65 billion US dollars"
doc = nlp(my_string)

for ent in doc.ents:
    print(ent.text, ent.label_)
```

**输出：**

```
('Google', 'ORG')
('Mountain View', 'GPE')
('California', 'GPE')
('109.65 billion US dollars', 'MONEY')
```

我们可以看到 spaCy 模型多么巧妙且自动地轻松识别出单词 **Google** 是一个 **ORG**，**California** 是一个 **GPE** 实体，并且在给定的句子中我们谈论的是 **109.65 billion US dollars**，这实际上是关于金钱的。

让我们尝试探索更多示例。

### 依存句法分析

依存句法分析是 spaCy 中更优美且强大的功能之一，它既快速又准确。该解析器还可用于句子边界检测，并允许你遍历基础名词短语，或称为“块”。

spaCy 的这一功能为你提供了一个解析树，解释了单词或短语之间的父子关系，且独立于单词出现的顺序。

让我们举个例子，假设你需要解析以下句子：

*帮我预订从班加罗尔到果阿的航班*

#### 示例 1：

```python
doc = nlp(u'Book me a flight from Bangalore to Goa')
blr, goa = doc[5], doc[7]

list(blr.ancestors)
```

输出：

```
[from, flight, Book]
```

上述输出告诉我们，用户希望预订从班加罗尔出发的航班。让我们尝试列出 **goa.ancestors** 对象的祖先：

```python
list(goa.ancestors)
```

输出：

```
[to, flight, Book]
```

这个输出告诉我们，用户希望预订前往果阿的航班。

#### 依存句法分析中的“祖先”是什么？

祖先是该词的句法后代中最右侧的标记。就像上面的例子中，对象 **blr** 的祖先是 *from*、*flight* 和 *Book*。

请记住，你始终可以使用 **ancestors** 属性列出 **doc** 对象项的祖先。

```python
list(doc[4].ancestors) #doc[4]==flight
```

上述代码将输出：

```
[flight, Book]
```

要以编程方式检查一个 **doc** 对象项是否是另一个 *doc* 对象项的祖先，我们可以这样做：

```python
doc[3].is_ancestor(doc[5])
```

上述代码返回 **True**，因为 **doc[3]**（即 flight）是 **doc[5]**（即 Bangalore）的祖先。你可以尝试更多这样的例子，以更好地理解依存句法分析和祖先的概念。

如果我们尝试思考一个在构建聊天机器人时可能实际遇到的真实场景，我们可能会遇到这样的句子：

> 我想预订一辆去酒店的出租车和一张餐厅的桌子。

在这个句子中，了解请求了哪些*任务*以及它们的目标是什么（即*用户是想预订去酒店的出租车，还是餐厅的桌子*）非常重要。

让我们尝试使用以下代码来做到这一点：

#### 示例 1：

```python
doc = nlp(u'Book a table at the restaurant and the taxi to the hotel')
tasks = doc[2], doc[8] #(table, taxi)

tasks_target = doc[5], doc[11] #(restaurant, hotel)

for task in tasks_target:
    for tok in task.ancestors:
        if tok in tasks:
            print("Booking of {} belongs to {}".format(tok, task))
            break
```

输出：

```
Booking of table belongs to restaurant
Booking of taxi belongs to hotel
```

#### 依存句法分析中的“子节点”是什么？

子节点是该标记的直接句法依赖项。我们可以像使用 **ancestors** 一样，使用 **children** 属性查看一个词的子节点。

```python
list(doc[3].children)
```

将输出

```

```

### 查找相似度

查找两个词之间的相似度是你在处理NLP时最常遇到的用例之一。有时，判断两个词是否相似变得相当重要。在构建聊天机器人时，你经常会遇到这样的情况：你不仅需要找到外观相似的词，还需要了解两个词在逻辑上的关联程度。

spaCy使用高质量的词向量，通过*GloVe算法*（全局词向量表示）来查找两个词之间的相似度。

GloVe是一种无监督学习算法，用于获取词的向量表示。GloVe算法使用从语料库中聚合的全局词-词共现统计信息来训练模型。

让我们尝试使用token的*vector*属性来查看spaCy向量中的实际值。

```python
doc = nlp(u'How are you doing today?')
for token in doc:
    print(token.text, token.vector[:5])
```

**输出：**

```
(u'How', array([-0.29742685, 0.73939574, -0.04001453, 0.44034013, 2.8967502], dtype=float32))
(u'are', array([-0.23435134, -1.6145049, 1.0197453, 0.9928169, 0.28227055], dtype=float32))
(u'you', array([0.10252178, -3.564711, 2.4822793, 4.2824993, 3.590245], dtype=float32))
(u'doing', array([-0.6240922, -2.0210216, -0.91014993, 2.7051923, 4.189252], dtype=float32))
(u'today', array([3.5409122, -0.62185854, 2.6274266, 2.0504875, 0.20191991], dtype=float32))
(u'?', array([2.8914998, -0.25079122, 3.3764176, 1.6942682, 1.9849057], dtype=float32))
```

看到这个输出，它并没有太多意义。从应用的角度来看，最重要的是不同词的向量有多相似——也就是说，词本身的含义。

为了在spaCy中查找两个词之间的相似度，我们可以这样做。

#### 示例 1：

```python
hello_doc = nlp(u"hello")

print(hello_doc.similarity(hella_doc))
```

#### 输出：

```

0.4193425861242359
```

如果你看*hello*这个词，它与*hi*这个词更相关、更相似，尽管*hello*和*hella*之间只差一个字符。

让我们再举一个句子的例子，学习spaCy如何进行相似度比较。还记得我们之前章节中的《权力的游戏》例子吗？我们将尝试一下，看看相似度。

#### 代码：

```python
GoT_str1 = nlp(u"When will next season of Game of Thrones be releasing?")

```

#### 输出：

```

```

正如我们在这个例子中看到的，两个句子之间的相似度约为79%，这足以说明两个句子非常相似，事实也确实如此。这可以帮助我们在构建聊天机器人时节省大量编写自定义代码的时间。因此，我们得出一个事实：spaCy使用词向量为我们提供了两个词之间有意义的相似度，而不仅仅是看它们的拼写或字母。

### Dialogflow简介

Dialogflow通过构建引人入胜的语音和文本对话界面（如语音应用和聊天机器人），为用户提供了与产品交互的新方法。

Dialogflow由AI驱动。它帮助你与用户在网站、移动应用、Google Assistant、Amazon Alexa、Facebook Messenger以及其他流行平台和设备上建立连接。

下图来自Dialogflow，展示了他们如何处理用户请求。

![图3-1. Dialogflow架构工作原理图](img/3dd6f3813d40b4b667f3d24d47212e59_170_0.png)

以下是处理过程：

- 用户与输入设备对话。
- 用户查询进入Dialogflow引擎。
- Dialogflow尝试识别意图。
- 根据意图，执行操作并从数据库返回数据。
- 响应返回给意图。
- 响应被转换为可操作的数据。
- 用户的信息请求被返回给输出设备。

Dialogflow中有一个代理的概念，最好将其描述为自然语言理解模块。这些模块可以包含在你的应用、产品或服务中，并将用户的自然请求转换为可操作的数据。当用户输入与代理中的某个意图匹配时，这种转换就会发生。

代理也可以被设计为以特定方式管理对话流程。这可以通过上下文、意图优先级、槽填充、职责以及通过webhook的履行来实现。

您一定想知道回调 URL、验证令牌和页面访问令牌是什么。让我们试着理解一下。

### 回调 URL

回调 URL 无非是一个公开可访问的 URL，Facebook 将向该 URL 发送来自您 Facebook 页面的任何实时请求。

假设您正尝试在 **OnlineEats** 上支付餐费，然后被重定向到银行的支付页面。现在，**OnlineEats** 必须向银行提供一个回调 URL，以便在支付完成后将用户重定向到该 URL。

这里 Facebook 不会进行任何重定向，而是会获取用户在页面聊天框中发送的所有消息，并将其 POST 到 webhook 或回调 URL。

现在，一旦我们在服务器上收到消息，我们就会进行意图分类和实体解析，然后制定您想要回复用户的内容。

### 验证令牌

验证令牌是在订阅被验证时发送到您端点的任意字符串。需要它的原因是确保我们的服务器知道该请求是由 Facebook 发出的，并且与我们刚刚配置的订阅相关。

假设其他人知道了您的 webhook 并冒充 Facebook 发送消息，那么 *verify_token* 就会派上用场，您将验证来源是否正确。基于此令牌，您也可以处理来自多个来源的 POST 请求，因为不同的来源会定义不同的令牌，但回调 URL 是相同的。

### 访问令牌

Facebook API 需要页面访问令牌来管理 Facebook 页面。它们对于每个页面、管理员和应用程序都是唯一的，并且有到期时间。

> 注意 请妥善保管回调 URL 和验证令牌，以便现在配置 webhook。

### 在 Rasa 中决定管道

管道不过是一组用于训练模型的算法。Rasa NLU 有两个广泛使用的管道，称为 **spacy_sklearn** 和 让我们了解一下这两个。

#### spacy_sklearn

- *spacy_sklearn* 管道利用来自 GloVe 算法或 Facebook AI 团队开发的名为 fastText 的算法的预训练词向量。

#### tensorflow_embedding

- *spacy_sklearn* 在某些场景下表现非常出色，例如，假设你有一句用户语句：“波士顿的天气怎么样？”当我们用相同的语句示例训练模型后，再让它预测“伦敦的天气怎么样？”的意图时，我们的模型现在足够智能，能够识别出“波士顿”和“伦敦”这两个词是相似的，并且它们属于相同的意图。

- 这个管道在数据集较小时非常有用。

- *tensorflow_embedding* 管道不像 spacy_sklearn 那样使用任何预训练的词向量，而是根据我们自己提供的数据集进行调整。

- *tensorflow_embedding* 管道的一个优点是，我们的词向量将完全符合我们的领域需求。

- 为了通过一个例子解释 *tensorflow_embedding* 的工作原理：在英语中，“play”这个词可能与“一项运动”或“一种娱乐或休闲活动”密切相关，而与“an act”这个词看起来非常不同。但在戏剧领域，“play”和“an act”是密切相关的，其中“play”指的是“剧作家创作的一种文学形式”。因此，非常有必要告诉我们的模型要学习特定领域的知识，而不是因为某些预训练模型而感到困惑。

### 创建配置文件

让我们再次使用Jupyter以与之前相同的方式创建一个JSON文件，并将其命名为config.json。让我们将其放在我们的数据文件夹之外（即，在我们的项目目录horoscope_bot中）。

### 编写Python代码来训练模型和进行预测

以下是我们使用data.json和config.json在Python中训练模型的代码，使用*tensorflow_embedding*管道。

```python
from rasa_nlu.training_data import load_data
from rasa_nlu.model import Trainer
from rasa_nlu import config
from rasa_nlu.model import Interpreter

def train_horoscopebot(data_json, config_file, model_dir):
    training_data = load_data(data_json)
    trainer = Trainer(config.load(config_file))
    trainer.train(training_data)
    model_directory = trainer.persist(model_dir, fixed_model_name='horoscopebot')
    return model_directory

def predict_intent(text):
    interpreter = Interpreter.load('./models/nlu/default/horoscopebot')
    return interpreter.parse(text)
```

在代码的第一部分，我们从rasa_nlu包中导入所有必要的库。然后我们定义了两个方法，分别名为*train_horoscopebot*和*predict_intent*，其中第一个方法根据给定的数据、config_file和model_directory（存储模型的位置）训练模型，而predict_intent方法使用rasa_nlu中的**Interpreter**模型加载预训练的模型文件，并使用户能够预测任何新的文本示例。

### 理解 Rasa 概念

在完全尝试在代码中使用它们之前，理解一些与 Rasa 相关的特定概念非常重要。在本节中，我们将学习 Rasa NLU 的一些重要且非常有用的概念。请确保你完全理解这些概念，因为我们将使用它们来构建第一个使用 Rasa 域文件格式的内部聊天机器人。如果我们不理解这些概念的含义，那么下一步将会很困难。

### Action（动作）

顾名思义，这是一个可以采取的特定动作。根据 Rasa 文档，它指的是*为响应对话而采取的动作*。

例如，如果用户询问今天的星座运势，我们的机器人可以执行“GetTodaysHoroscope”动作。让我们看看一个“GetTodaysHoroscope”动作以代码形式会是什么样子。

```python
from rasa_core.actions import Action
from rasa_core.events import SlotSet

class ActionGetTodaysHoroscope(Action):
    def name(self):
        return "get_todays_horoscope"

    def run(self, dispatcher, tracker, domain):
        # type: (Dispatcher, DialogueStateTracker, Domain) -> List[Event]
        user_horoscope_sign = tracker.get_slot('horoscope_sign')
        """Write your logic to get today's horoscope details
        for the given Horoscope sign based on some API calls or retrieval from the database"""
        return [SlotSet("horoscope_sign", user_horoscope_sign)]
```

**name** 方法返回动作的名称，我们将在域文件中将其作为自定义方法名来引用。

**run** 方法执行完成动作的主要工作——即核心业务逻辑就在这里。如你所见，它接受三个参数：*dispatcher*、*tracker* 和 *domain*。

让我们逐一理解这些参数：

- **dispatcher**：调度器用于向我们的用户发送消息。我们可以使用 **dispatcher.utter_message()** 来实现这一点。
- **tracker**：当前用户的状态跟踪器。我们可以使用 **tracker.get_slot(slot_name)** 来访问槽值，要获取最近的用户消息，我们可以使用 **tracker.latest_message**。
- **domain**：机器人的域。我们将在本章稍后更详细地讨论域。

> **注意：** run 方法返回一个事件实例列表。

### SLOTS（槽）

槽是使机器人能够像人类一样工作的关键。槽就像一个存储空间，可以存储用户提供的信息，也可以使用从数据库或 API 预先获取的信息。

不同的用例有不同的槽类型：

例如，在我们想要构建一个星座机器人的用例中，我们可能希望将用户提供的 **horoscope_sign** 的槽类型设置为文本。

根据你想要使用的槽类型，Rasa 提供了一些预定义的槽类型。

除了文本，Rasa 还有以下槽类型：

- **Boolean** ⇒ 用于真/假值
- **Categorical** ⇒ 用于需要从多个值中选择一个的情况
- **Float** ⇒ 用于连续值
- **List** ⇒ 用于值列表
- **Featurized** ⇒ 用于不影响对话的内部值存储

### Templates（模板）

模板这个词你一生中至少听过一次，无论是寻找发送电子邮件的模板、准备文档的模板、构建作品集网站的模板，还是遵循流程的模板。

Rasa 中的模板用于话语。话语模板包含一组预设文本，当某个动作被触发时发送给用户。通过将动作名称与话语名称相同，或者通过带有自定义代码的动作，我们可以将格式化的消息通过模板发送给用户。

域文件中模板的一个简单表示可能如下所示：

```yaml
templates:
  utter_greet:
    - "hello {name} can be filled by a slot of same name or by custom code"
  utter_goodbye:
    - "goodbye"
    - "take care multiple templates allow the bot to randomly pick from them"
  utter_default:
    - "Sorry, I didn't get that."
```

现在，你明白为什么我们之前需要通过理解 Rasa 核心概念来为这种情况做准备了。

让我们用 YAML 定义创建一个 DefaultDomain 文件。Rasa 使用 .yml 文件来定义域格式。

最初 YAML 据说是 **Yet Another Markup Language**（又一种标记语言），描述了它作为标记语言的用途，但后来改为理解为 YAML Ain't Markup Language（YAML 不是标记语言），这是一个递归首字母缩略词，以区别其作为数据导向而非文档标记语言的目的。

如我们在示例中所见，主要定义了五个动作，其中前三个动作只是向用户说出模板文本，但最后两个动作要求我们从数据库中检索数据或调用 API 获取当天的星座运势并返回给用户。

在 **subscribe_user** 动作的情况下，我们也需要执行一个操作，将当前用户添加到数据库的订阅列表中。这些用户定义的动作称为自定义动作。要拥有这样的自定义动作，我们需要编写当这些动作被触发时机器人应该做什么。

### 编写聊天机器人的自定义动作

我们知道，每当一个 **UtterAction** 被触发时，我们的机器人将用为该动作在模板中定义的话语文本进行响应。但是当某个自定义动作被触发时会发生什么？在本节中，我们将编写 Python 代码来创建自定义动作，我们可以用它们来调用 API 或者几乎任何你可以用 Python 做的事情。

让我们在项目目录中创建一个新文件，称为 **actions.py**（即在我们的例子中是在 **horoscope_bot** 文件夹内）。

```python
from __future__ import absolute_import
from __future__ import division
from __future__ import print_function
from __future__ import unicode_literals

import requests

from rasa_core_sdk import Action
from rasa_core_sdk.events import SlotSet
```

class GetTodaysHoroscope(Action):

    def name(self):
        return "get_todays_horoscope"

    def run(self, dispatcher, tracker, domain):
        # type: (Dispatcher, DialogueStateTracker, Domain) -> List[Event]
        user_horoscope_sign = tracker.get_slot('horoscope_sign')

        base_url = "http://horoscope-api.herokuapp.com/horoscope/{day}/{sign}"
        url = base_url.format(**{'day': "today", 'sign': user_horoscope_sign})

        # http://horoscope-api.herokuapp.com/horoscope/today/capricorn
        res = requests.get(url)

        todays_horoscope = res.json()['horoscope']

        response = "Your today's horoscope:\n{}".format(todays_horoscope)

        dispatcher.utter_message(response)

        return [SlotSet("horoscope_sign", user_horoscope_sign)]

正如我们所见，在我们的名为 GetTodaysHoroscope 的 action 中有两个方法。第一个方法 **name** 只是返回该 action 的名称。另一个方法是 **run**，正如之前讨论的，它是通过执行我们编写的业务逻辑来实际完成任务的方法。

在我们的方法中，我们利用了一个开源 API，其代码托管在 GitHub 上。该 API 的 URL 如下所示：

[http://horoscope-api.herokuapp.com/horoscope/today/capricorn](http://horoscope-api.herokuapp.com/horoscope/today/capricorn)

它返回的数据格式为 JSON：

```json
{
    "date": "2018-08-29",
    "horoscope": "You will be overpowered with nostalgia and may long to get in touch with old pals. And as Ganesha says, chances are that you may take a liking to your ex-lover, while simultaneously strengthening your social standing. All in all, the day will be a productive one.",
    "sunsign": "capricorn"
}
```

正如你在 **run** 方法中看到的，我们将 API 的响应转换为 Python JSON 对象，然后从 JSON 中访问 ‘horoscope’ 键以获取实际的星座运势。从 JSON 中获取实际的星座运势后，我们使用 **dispatcher** 对象及其方法 utter_message 来构建响应并将其发送回用户。

最后，我们使用 **SlotSet** 方法设置槽位。SlotSet 就像是保存你从用户响应中提取出的变量，以便你可以在对话流程中的任何时间在代码中使用它们。

**注意：使用上述 API，我们可以通过提供星座符号来获取今日的星座运势。你可以自由使用自己的 API 或数据库。你只需要将 API 调用替换为你想要使用的其他数据源。**

就像我们在 **actions.py** 文件中添加了 **GetTodaysHoroscope** action 一样，我们也将添加 **SubscribeUser** action。我们不会使用任何数据库来存储用户的订阅偏好，但当你为真实用户构建聊天机器人时，你可能需要拥有 user_ids，以便在数据库中将其与他们的订阅关联起来。

以下是 SubscribeUser action 的样子：

```python
class SubscribeUser(Action):
    def name(self):
        return "subscribe_user"

    def run(self, dispatcher, tracker, domain):
        # type: (Dispatcher, DialogueStateTracker, Domain) -> List[Event]
        subscribe = tracker.get_slot('subscribe')

        if subscribe == "True":
            response = "You're successfully subscribed"
        if subscribe == "False":
            response = "You're successfully unsubscribed"

        dispatcher.utter_message(response)

        return [SlotSet("subscribe", subscribe)]
```

这样，我们可以根据需要编写任意数量的自定义 action。

让我们在主项目目录中创建一个名为 **train_initialize.py** 的新文件。**train_initialize.py** 的内容如下所示：

```python
from future import absolute_import
from future import division
from future import print_function
from future import unicode_literals

from rasa_core import utils
from rasa_core.agent import Agent
from rasa_core.policies.keras_policy import KerasPolicy
from rasa_core.policies.memoization import MemoizationPolicy
from rasa_core.policies.sklearn_policy import SklearnPolicy

if __name__ == '__main__':
    utils.configure_colored_logging(loglevel="DEBUG")
    training_data_file = './data/stories.md'
    model_path = './models/dialogue'
    agent = Agent("horoscope_domain.yml",
                  policies=[MemoizationPolicy(), KerasPolicy()])
    training_data = agent.load_data(training_data_file)
    agent.train(
        training_data,
        augmentation_factor=50,
        epochs=500,
        batch_size=10,
        validation_split=0.2
    )

```

这就是我们在 **train_initialize.py** 文件中编写的代码。现在，在进入下一个代码文件之前，让我们先尝试理解其中的要点。

首先，我们从 future 模块导入一些方法。Python 中的 future 语句有特殊用途，它们会改变 Python 模块的解析方式，并改变现有方法的行为方式。

从 rasa_core 模块导入 **utils** 方法以配置日志记录。从 agent 模块导入 agent 类以创建代理对象。KerasPolicy、MemorizationPolicy 将作为策略参数传递给 agent 类。**configure_colored_logging**：在 utils.py 中定义的实用方法，使用 Python 的 coloredlogs 包进行彩色日志记录。

**Agent**：Rasa 定义的一个类，提供了一个接口来利用最重要的 Rasa Core 功能，例如训练、处理消息、加载对话模型、获取下一个动作以及处理通道。

**load_data**：从给定路径加载训练数据。
**train**：使用提供的文件中的数据训练给定的策略/策略集成。
**training_data**：由 **load_data** 方法返回的对象。这实际上就是我们的训练数据文件。
**augmentation_factor**：告诉 Rasa 根据我们的初始故事集应该创建多少个虚拟故事。10 倍因子是训练数据生成器增强轮次的经验值。
**epochs**：1 个 epoch 是在整个训练数据集上完成的一个完整训练周期。训练数据的前向和后向传递的总次数。
**batch_size**：告诉你每次传递中使用的训练样本数量。100 个样本，**batch_size** 为 10，需要 10 个 epoch 才能遍历整个数据集。
**validation_split**：用于验证模型无偏准确率的数据百分比。
**persist**：此方法用于将代理对象持久化到给定目录中以供重用。

### 通过强化进行实时训练

在本节中，我们将编写更多代码来训练我们的对话模型，并在它给出错误输出时重新训练它。

因此，当我们的机器人做错事时，我们立即介入，通过告诉模型什么是正确的，让它知道其预测是错误的。无需停止，模型会重新训练自己，一旦我们完成，用户和机器人之间的交互就会被捕获到一个文件中，并添加到我们现有的训练数据中。它更像是一个在每一步都提供反馈的系统，而不是等待最后的单一奖励。

```python
from future import absolute_import
from future import division
from future import print_function
from future import unicode_literals

import logging

from rasa_core import utils, train
from rasa_core.training import online
from rasa_core.interpreter import NaturalLanguageInterpreter

logger = logging.getLogger(__name__)

def train_agent(interpreter):
    return train.train_dialog_model(
        domain_file="horoscope_domain.yml",
        stories_file="data/stories.md",
        output_path="models/dialog",
        nlu_model_path=interpreter,
        endpoints="endpoints.yml",
        max_history=2,
        kwargs={
            "batch_size": 50,
            "epochs": 200,
            "max_training_samples": 300
        }
    )

if __name__ == '__main__':
    utils.configure_colored_logging(loglevel="DEBUG")
    nlu_model_path = "./models/nlu/default/horoscopebot"
    interpreter = NaturalLanguageInterpreter.create(nlu_model_path)
    agent = train_agent(interpreter)
    online.serve_agent(agent)
```

**max_history** 是模型需要跟踪的状态数量。

### Rasa 的凭证管理

Rasa 提供了一种在一处管理所有凭证的方法。你可能有一个单一的模型，但你可能希望将其部署到 Facebook、Slack、Telegram 等各种其他平台上。所有这些第三方平台在连接时都需要一些凭证。这些凭证存储在一个名为的 YAML 文件中。

让我们在我们的项目目录 **horoscope_bot** 文件夹中创建一个名为 **credentials.yml** 的文件，并在其中添加我们的 Facebook 凭证。如果你不知道如何获取，现在只需创建文件，在本章的下一节中你可以找到获取 Facebook 凭证的步骤。

**credentials.yml** 的内容将如下所示：

```
facebook:

verify: "horoscope-bot"

secret: "bfe5a34a8903e745e32asd18"

page-access-token:
"HPaCAbJJ1JmQ7qDedQKdjEAAAbO4ijKr7H9nx4rEBAAuFk4Q3gPQcNTowtD"
```

这些是虚拟凭证；令牌或密钥的长度以及字符类型可能因你的 Facebook 应用程序而异。

如果你正在处理一个大型项目，需要将聊天机器人集成到各种平台上，并且希望使项目更易于维护，那么最好使用。如果你是一家企业，试图构建一个在 Facebook、Slack、Twitter、Telegram 或你自己的网站等各种平台上以相同方式工作的机器人，我强烈建议你维护一个 **credentials.yml**。在这种情况下，管理密钥和密钥会变得更容易。

管理应用级密钥的一个好方法是将密钥存储为环境变量，并编写代码从操作系统环境本身读取密钥值或任何其他敏感信息。请记住，将任何类型的密钥保存在代码中绝不是一个好主意。

你也可以在服务器上创建一个点（.）env文件，并从该文件中读取密钥，该文件不会在代码仓库的任何地方被跟踪。

为了简单起见，我们将在用于部署的独立脚本中使用访问密钥和密钥。我们将使其易于理解，以便你首先能够构建机器人，然后你可以尝试扩展它，最重要的是，你可以考虑安全级别问题。

如果你需要将机器人部署到多个平台，并希望使用credentials.yml来维护不同的凭证，那么你可以通过传递一个额外的参数来使用它。例如，要在运行rasa core时使用上述名为**credentials.yml**的凭证文件，你可以使用以下命令。

```
python -m rasa_core.run -d models/dialogue -u models/nlu/current
—port 5002—credentials credentials.yml
```

### 为你的聊天机器人通道编写脚本

这个脚本与我们迄今为止所学和所写的类似，但它需要我们覆盖 `rasa_core` 的一些现有方法，以便我们可以定义自己的 API 认证规则。我在下面的代码中进行了基本的字符串检查以进行令牌验证。这不建议用于生产级系统，因此如果你正在为大型系统构建聊天机器人，请务必仔细编写该部分。

创建一个名为 `myown_chatbot.py` 的新文件，并将以下内容添加到其中：

```python
import os

from rasa_core.channels.rasa_chat import RasaChatInput
from rasa_core.agent import Agent

from rasa_core.interpreter import RasaNLUInterpreter
from rasa_core.utils import EndpointConfig

# load your trained agent
interpreter = RasaNLUInterpreter("models/nlu/default/horoscopebot/")
MODEL_PATH = "models/dialogue"

action_endpoint = EndpointConfig(url="https://horoscopebot1212-actions.herokuapp.com/webhook")

agent = Agent.load(MODEL_PATH, interpreter=interpreter,
                   action_endpoint=action_endpoint)

class MyNewInput(RasaChatInput):
    @classmethod
    def name(cls):
        return "rasa"

    def _check_token(self, token):
        if token == 'mysecret':
            return {'username': 1234}
        else:
            print("Failed to check token: {}.".format(token))
            return None

input_channel = MyNewInput(url='https://horoscopebot1212.herokuapp.com')  # set serve_forever=False if you want to keep the server running

s = agent.handle_channels([input_channel], int(os.environ.get('PORT', 5005)), serve_forever=True)
```

这里有几个要点需要注意：

- `rasa_core` 中的 `_check_token` 方法基本上如下所示，它会进行 API 调用以获取用户对象。这主要完成用户级别的认证工作。在前面覆盖的方法中，我们保持简单以使其工作并理解其用法。

```python
def _check_token(self, token):
    url = "{}/users/me".format(self.base_url)
    headers = {"Authorization": token}

    logger.debug("Requesting user information from auth server {}.".format(url))

    result = requests.get(url, headers=headers, timeout=DEFAULT_REQUEST_TIMEOUT)

    if result.status_code == 200:
        return result.json()

    logger.info("Failed to check token: {}. Content: {}".format(token, result.data))
    return None
```

- 使用 Rasa 自己的 `_check_token` 方法可能需要你编写一个 API 或 Web 服务，该服务接受请求并以指定方式返回响应。
- 确保将动作服务器端点更改为你自己的 URL。
- 请记住，代码中的 `mysecret` 字符串将用于后续进行 API 调用。
